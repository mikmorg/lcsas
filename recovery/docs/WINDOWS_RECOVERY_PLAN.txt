LCSAS RECOVERY -- WINDOWS RECOVERY PATH (PLAN)
================================================

GOAL

Provide a recovery workflow on a stock Windows 10/11 laptop that
requires NO additional installation (no WSL, no Python, no Cygwin,
no admin rights for the common case).

Target user: an average inheritor with a Windows laptop, a USB BD
reader, the meta-volume disc, and the password.  No technical
expertise beyond "double-click an icon" and "type a password".

TARGET PATFORMS (in priority order)
  1. Windows 10/11 on x86_64 (>95% of Windows market)
  2. Windows 11 on ARM64 (laptops with Snapdragon X / Surface Pro X)
  3. Older Windows back to Windows 7 SP1 (best-effort; no testing)

────────────────────────────────────────────────────────────────────
PART 1 -- SOURCE PORTABILITY (lcsas-restore.exe)
────────────────────────────────────────────────────────────────────

CROSS-COMPILER

Use `zig cc` with `-target x86_64-windows-gnu`.  Rationale:

  * Single binary download (~50 MB).  Stable releases.  Bundles its
    own MinGW-w64 headers; no separate toolchain install.
  * Deterministic output.  Already part of recovery/Makefile (for
    aarch64/riscv64 Linux targets).
  * Adding a Windows target is one more Makefile entry.

Alternative (documented in BUILD.txt as a fallback path):
  * `x86_64-w64-mingw32-gcc` from a standard MinGW-w64 install.

PORTABILITY CHANGES TO C SOURCE

The hand-written code is mostly portable already (sha256, aes,
poly1305, scrypt, pbkdf2, b64, hex, json_q, path).  The Windows
delta is concentrated in the I/O wrappers.

Create a single header `src/lcsas-restore/posix_compat.h`:

    #ifndef LCSAS_POSIX_COMPAT_H
    #define LCSAS_POSIX_COMPAT_H

    #ifdef _WIN32
        #include <io.h>            /* open, close, read, write, lseek */
        #include <direct.h>        /* _mkdir */
        #include <fcntl.h>
        #include <sys/stat.h>
        #include <sys/types.h>

        /* mkdir is one-arg on Windows; drop the mode. */
        static int posix_mkdir(const char *p, int mode) {
            (void)mode; return _mkdir(p);
        }
        #define mkdir(p, m)  posix_mkdir((p), (m))

        /* Windows uses backslash; we always pass forward slashes
         * which the runtime accepts.  No path translation needed. */

        /* No symlink support without privileges.  Provide a stub
         * that errors out; tree.c handles this gracefully. */
        static int symlink(const char *target, const char *link) {
            (void)target; (void)link;
            errno = EPERM;
            return -1;
        }

        /* chmod is a no-op on Windows. */
        #define chmod(p, m)  ((void)(p), (void)(m), 0)

        /* fsync, fdatasync: best-effort via _commit. */
        #define fsync(fd)    _commit(fd)

        /* ssize_t is not in MinGW's standard headers in all configs. */
        #ifndef _SSIZE_T_DEFINED
        typedef long long ssize_t;
        #define _SSIZE_T_DEFINED
        #endif

    #else
        #include <unistd.h>
        #include <fcntl.h>
        #include <sys/stat.h>
        #include <dirent.h>
    #endif

    #endif

Source files to update:
  * io.c        -- replace direct unistd/sys-stat includes with
                   #include "posix_compat.h"
  * repo.c      -- same
  * tree.c      -- same; also handle symlink() returning -1 with
                   EPERM (log warning instead of error)
  * iso9660.c   -- already pure POSIX I/O; same treatment

DIRENT.H

MinGW-w64 ships its own <dirent.h> that supports opendir / readdir
/ closedir with the POSIX struct layout.  No changes needed for
repo.c's directory walks.

UTF-8 / WIDE-CHAR PATHS

MinGW's POSIX shims operate on MBCS / current-codepage strings, not
UTF-8 by default.  For Windows 10 1903+, the application manifest
can opt into UTF-8 for the entire process:

  https://learn.microsoft.com/en-us/windows/apps/design/globalizing/
       use-utf8-code-page

Embed an app manifest that sets `<activeCodePage>UTF8</activeCodePage>`.
This lets our existing UTF-8-handling code (e.g. JSON-decoded paths)
just work with `open()` / `mkdir()` on Windows ≥10 1903.

For older Windows: document in RECOVER_WINDOWS.txt that non-ASCII
filenames may be mangled.

64-BIT FILE OFFSETS

MinGW's `lseek` is 32-bit; we need `_lseeki64`.  Add to posix_compat.h:

    #ifdef _WIN32
    #define lseek(fd, off, whence)  _lseeki64((fd), (off), (whence))
    typedef long long off_t;
    #endif

This matters: pack files routinely exceed 4 GiB; without
_lseeki64 they'd silently truncate.

PART 2 -- BUILD SYSTEM
────────────────────────────────────────────────────────────────────

MAKEFILE TARGETS

Add to recovery/Makefile:

    bin/x86_64-windows/lcsas-restore.exe: ARCH=x86_64-windows
    bin/x86_64-windows/lcsas-restore.exe:
        $(MAKE) clean
        $(MAKE) BUILD=build/win64 \
                CC="python3 -m ziglang cc -target x86_64-windows-gnu" \
                CFLAGS_EXTRA="-D_WIN32_WINNT=0x0601 -municode" \
                LDFLAGS="-static -lws2_32"
        mkdir -p bin/x86_64-windows
        cp build/win64/lcsas-restore.exe bin/x86_64-windows/

    bin/aarch64-windows/lcsas-restore.exe: ARCH=aarch64-windows
    bin/aarch64-windows/lcsas-restore.exe:
        $(MAKE) clean
        $(MAKE) BUILD=build/winarm64 \
                CC="python3 -m ziglang cc -target aarch64-windows-gnu" \
                LDFLAGS="-static"
        mkdir -p bin/aarch64-windows
        cp build/winarm64/lcsas-restore.exe bin/aarch64-windows/

EXE EXTENSION

Update the existing rule `$(BUILD)/lcsas-restore` to emit a `.exe`
suffix on Windows targets:

    BIN_EXT =
    ifeq ($(findstring windows,$(CC)),windows)
        BIN_EXT = .exe
    endif

    $(BUILD)/lcsas-restore$(BIN_EXT): ...

REPRODUCIBILITY

zig cc honours SOURCE_DATE_EPOCH and `-Wl,--no-insert-timestamp`.
Add to Windows LDFLAGS:

    LDFLAGS = -static -Wl,--no-insert-timestamp

────────────────────────────────────────────────────────────────────
PART 3 -- DRIVER SCRIPT (restore.bat / restore.cmd)
────────────────────────────────────────────────────────────────────

Windows users cannot run POSIX-sh natively (no /bin/sh, no
bash.exe by default in stock Windows).  We need a CMD-compatible
driver script: `recovery/scripts/restore.bat`.

The .bat must replicate the cascade (tiers 1, 2, 3-skip, 4-skip,
5):

    @echo off
    setlocal enabledelayedexpansion

    REM Detect arch (PROCESSOR_ARCHITECTURE).
    if /i "%PROCESSOR_ARCHITECTURE%"=="AMD64" set ARCH=x86_64-windows
    if /i "%PROCESSOR_ARCHITECTURE%"=="ARM64" set ARCH=aarch64-windows

    REM Discover RECOVERY root.
    set SCRIPT_DIR=%~dp0
    if exist "%SCRIPT_DIR%..\bin" set RECOVERY=%SCRIPT_DIR%..
    if exist "%SCRIPT_DIR%recovery\bin" set RECOVERY=%SCRIPT_DIR%recovery

    REM Tier 1: prebuilt lcsas-restore.exe.
    set BIN=%RECOVERY%\bin\%ARCH%\lcsas-restore.exe
    if exist "%BIN%" goto :do_restore

    REM Tier 2: rustic-static.exe.
    set BIN=%RECOVERY%\bin\%ARCH%\rustic-static.exe
    if exist "%BIN%" goto :do_rustic

    REM Tier 3 (rebuild) and tier 4 (cargo) are not supported on
    REM Windows without an installed toolchain; skip silently.

    REM Tier 5: Python fallback (only if py + standalone_restorer.py).
    where py >nul 2>nul && (
        if exist "%RECOVERY%\..\standalone_restorer.py" (
            set BIN=py
            goto :do_python
        )
    )

    echo ERROR: no recovery method available on this Windows machine.
    echo Looked for:
    echo   %RECOVERY%\bin\%ARCH%\lcsas-restore.exe
    echo   %RECOVERY%\bin\%ARCH%\rustic-static.exe
    echo See %RECOVERY%\docs\RECOVER_WINDOWS.txt for manual options.
    pause
    exit /b 1

    :do_restore
    set /p TARGET=Restore target folder [C:\Users\%USERNAME%\Documents\restored]:
    if "!TARGET!"=="" set TARGET=C:\Users\%USERNAME%\Documents\restored
    mkdir "!TARGET!" 2>nul
    set /p PASSWORD=Password:
    > "%TEMP%\lcsas-pw.txt" echo !PASSWORD!
    "%BIN%" --repo "%RECOVERY%\repo" --password-file "%TEMP%\lcsas-pw.txt" --target "!TARGET!" --snapshot latest
    del "%TEMP%\lcsas-pw.txt"
    pause
    exit /b %ERRORLEVEL%

    :do_rustic
    REM analogous, rustic CLI args
    ...

    :do_python
    REM analogous, py + standalone_restorer.py
    ...

DESIGN NOTES

* `pause` at the end so the console doesn't auto-close when the
  user double-clicks the .bat (otherwise they see nothing).
* `set /p` to prompt for path + password.  Password is visible
  while typing (CMD has no `read -s` equivalent without PowerShell).
  For privacy, document a PowerShell alternative.
* Password file is written to %TEMP% and deleted after; not ideal
  but acceptable for the common case.
* The .bat MUST be CRLF line endings; commit with `* text eol=crlf`
  in .gitattributes for *.bat files.

    [ ] Insert ISO of meta-volume.

    [ ] Recovery completes; files in Documents\restored.

PART 7 -- KNOWN LIMITATIONS / NON-GOALS
────────────────────────────────────────────────────────────────────

OUT OF SCOPE for the Windows recovery path:

  * Symlink restoration.  Windows symlinks require SeCreateSymbolic
    LinkPrivilege; we don't try to acquire it.  Symlinks become a
    text file containing the target path (a marker the user can
    inspect manually).
  * Owner/group/UID/GID restoration.  Windows doesn't have these.
  * Extended attributes.  Windows alternate-data-streams are not
    populated.
  * Hardlinks across volumes.  Windows hardlinks need same-volume
    semantics; we silently copy instead.
  * UEFI boot from disc on Windows hardware.  Windows boot loader
    cannot chain to ISO9660 directly; this is a meta-volume-only
    issue.  Workaround: user boots a Linux live USB if they need
    to boot from disc.

NON-GOALS

  * A native GUI launcher.  Phase 5+; .bat console UX is the
    common-case answer.
  * Codesigning the .exe.  Authenticode certificates expire every
    1-3 years and cost money; long-term archival is incompatible
    with cert-rotation.  SmartScreen will show a warning; documented.
  * Anti-virus whitelisting.  Same reasoning.

────────────────────────────────────────────────────────────────────
SUCCESS CRITERIA
────────────────────────────────────────────────────────────────────

A Windows 10/11 user, given the meta-volume disc and the password,
can recover their files by:

  1. Plugging in a USB BD reader.
  2. Inserting the disc.
  3. Opening File Explorer to the disc.
  4. Double-clicking restore.bat.
  5. Typing the password.
  6. Waiting.

No additional installation, no admin rights, no terminal expertise.

Verified by Wine-based automated test in CI; manually re-verified
on a real Windows 10/11 VM at each major release.
