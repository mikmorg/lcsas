LCSAS RECOVERY ON WINDOWS -- STEP BY STEP
============================================

WHO THIS IS FOR

You are running Windows 10 (version 1709 / October 2017 or later)
or Windows 11.  You have inherited a collection of LCSAS recovery
discs and you want to read the files on them.  You have the
password for the archive.

WHY 1709?

  Our prebuilt lcsas-restore.exe imports the Windows Universal C
  Runtime (UCRT, the api-ms-win-crt-*.dll family).  UCRT ships in
  the OS starting with Windows 10 1709.  Windows 7 SP1 / 8 / 8.1
  also have it IF Windows Update KB2999226 (Sep 2015) was installed;
  most patched systems do.

  Windows XP / Vista / unpatched 7-8 cannot run the bundled binary.
  For those systems, see "ON OLDER WINDOWS" below.

You do NOT need:

  * Admin rights.
  * To install WSL, Cygwin, or any developer tools.
  * To re-partition your disk.
  * To boot from the disc.

You DO need:

  * A USB Blu-ray reader (or any optical drive that reads the disc
    format your archive uses).
  * The password.

────────────────────────────────────────────────────────────────────
QUICK START (90 SECONDS)
────────────────────────────────────────────────────────────────────

  1. Plug in your USB Blu-ray reader.
  2. Insert the disc labelled "LCSAS_META" (or any meta-volume disc).
  3. Open File Explorer.  The disc should appear under "This PC" as
     a drive (typically D:, E:, or F:).
  4. Open the disc.
  5. Double-click  restore.bat.
  6. A black console window opens.  When prompted:
        - press Enter to accept the default restore folder
          (Documents\restored), OR type a folder of your choosing.
        - type the password and press Enter.
  7. Wait.  When the script says "Recovery complete", the files are
     in the folder you chose.

If you have multiple discs, the script will tell you when to swap.

You do NOT need two Blu-ray drives.  Before the prompt, restore.bat
copies itself and the binary tree from the meta-disc into %TEMP%
and re-launches from there.  After that copy, no file handles
remain on the meta-disc, so you can eject the recovery disc and
insert each data disc into the same drive when prompted.

────────────────────────────────────────────────────────────────────
VERIFY THE BINARY BEFORE RUNNING IT (recommended for paranoid users)
────────────────────────────────────────────────────────────────────

Every file on the disc has a SHA-256 hash recorded in
recovery\MANIFEST.sha256.  Before double-clicking restore.bat, you
can verify both the script and the binary against the manifest
using Windows' built-in certutil tool.  No third-party software
required.

  STEP 1.  Open a Command Prompt and change to the disc.

      D:                              (or whichever drive letter)
      cd D:\

  STEP 2.  Compute the SHA-256 of restore.bat and lcsas-restore.exe.

      certutil -hashfile restore.bat SHA256
      certutil -hashfile recovery\bin\x86_64-windows\lcsas-restore.exe SHA256

  STEP 3.  Look up the expected hash in recovery\MANIFEST.sha256:

      findstr /C:"restore.bat" recovery\MANIFEST.sha256
      findstr /C:"lcsas-restore.exe" recovery\MANIFEST.sha256

  The two values for each file must match.  If they do not, the
  disc has been tampered with or is corrupted -- do not run.

────────────────────────────────────────────────────────────────────
SMARTSCREEN WARNING ("Windows protected your PC")
────────────────────────────────────────────────────────────────────

When you double-click restore.bat for the first time, Windows
SmartScreen may show:

    Windows protected your PC
    Microsoft Defender SmartScreen prevented an unrecognized app
    from starting.

This is expected: the recovery tools are unsigned (signing
certificates expire on a 1-3 year cycle and are incompatible with
long-term archival).

To proceed:

  1. Click "More info".
  2. Click "Run anyway".

If your antivirus quarantines lcsas-restore.exe, restore it from
the antivirus's quarantine UI.  The binary's SHA-256 is recorded in
recovery\MANIFEST.sha256 on the disc; you can verify by hand:

    certutil -hashfile recovery\bin\x86_64-windows\lcsas-restore.exe SHA256

────────────────────────────────────────────────────────────────────
THE PASSWORD IS VISIBLE WHILE TYPING (CMD limitation)
────────────────────────────────────────────────────────────────────

Plain CMD has no equivalent of `read -s`.  If you want a hidden
password prompt, open PowerShell instead and run:

    $env:LCSAS_PASSWORD = (Read-Host -AsSecureString -Prompt "Password" | `
        ConvertFrom-SecureString -AsPlainText)
    D:\restore.bat

(Replace D: with your disc's drive letter.)

For non-interactive use (e.g. scripting), put the password in a
file and set the LCSAS_PWFILE environment variable:

    set LCSAS_PWFILE=C:\path\to\password.txt
    D:\restore.bat

The password file should contain only the password, no trailing
newline.

────────────────────────────────────────────────────────────────────
DESTINATION FOLDER NOTES
────────────────────────────────────────────────────────────────────

  * The default destination is %USERPROFILE%\Documents\restored,
    typically C:\Users\<you>\Documents\restored.

  * If the original archive has more data than you have free disk
    space, the restore will fail partway through.  Plan ahead -- the
    summary message at the start of the run includes the recovery
    root location but does not yet predict total size.  A future
    version of the tool will add --estimate-size.

  * Long paths: Windows historically limited paths to 260 characters.
    Modern Windows 10/11 supports longer paths if enabled in Group
    Policy.  Restored files with paths > 260 chars will fail to
    write.  Workaround: enable long-path support
    (Computer\HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\
     FileSystem\LongPathsEnabled = 1) or restore to a folder with a
    short root path like C:\r.

  * Symlinks in the original archive are NOT restored on Windows.
    Creating Windows symlinks requires the SeCreateSymbolicLink
    privilege which we do not request.  The corresponding restored
    paths will simply be missing.  Affected paths are reported on
    stderr.

  * Filename case: Windows is case-insensitive.  If the original
    archive contained two files differing only by case (foo.txt and
    FOO.TXT), only one will survive the restore.  This is rare in
    practice on archives created from Linux/macOS users' home dirs.

────────────────────────────────────────────────────────────────────
WHAT IF restore.bat DOES NOT RUN?
────────────────────────────────────────────────────────────────────

CASE A.  Double-click opens Notepad instead of running the script.
         File Explorer is set to "show extensions for known file
         types".  Right-click restore.bat -> Open with -> "Windows
         Command Processor".  Or open CMD/PowerShell and type
         `D:\restore.bat`.

CASE B.  The console window flashes and disappears.
         Open CMD/PowerShell first, navigate to the disc
         (`cd D:`), then run `restore.bat` so the error message
         stays on screen.

CASE C.  "lcsas-restore.exe is not recognized as an internal or
         external command."
         restore.bat looks for the .exe under
         recovery\bin\x86_64-windows\.  If that path is empty on
         your disc, the disc was built without the Windows binary.
         Options:

           (a) Cross-build on another machine and copy the .exe in.
               From a Linux/macOS host with the LCSAS source:
                   cd recovery
                   make windows
                   cp bin/x86_64-windows/lcsas-restore.exe \
                      /path/to/disc/recovery/bin/x86_64-windows/

           (b) Use WSL.  Install WSL2 from the Microsoft Store (no
               admin needed in recent Windows versions), then:
                   wsl
                   sh /mnt/d/recovery/scripts/restore.sh /mnt/d ~/restored
               (replace /mnt/d with your disc mount point inside WSL).

           (c) Use the Python fallback.  Install Python from
               python.org, then:
                   py D:\standalone_restorer.py D:\repo C:\Users\me\restored

CASE D.  Recovery completes but some files appear truncated.
         File a bug report with the SHA-256 of the affected file
         and the disc serial number.  This should not happen --
         every blob is verified against its SHA-256 hash before
         being written.  A truncation almost certainly means a
         bug in the tool.

────────────────────────────────────────────────────────────────────
PRIVACY / SECURITY
────────────────────────────────────────────────────────────────────

  * The password is briefly written to %TEMP% and deleted after the
    binary exits.  If your machine has full-disk encryption, this is
    safe enough.  On shared machines, prefer the LCSAS_PWFILE
    workflow with a file on removable media.

  * The .exe is fully offline.  It makes no network connections.
    You can run it on an air-gapped machine.

  * Microsoft Defender will scan the binary on first run; this is
    normal and adds a few seconds.

────────────────────────────────────────────────────────────────────
TROUBLESHOOTING -- error messages
────────────────────────────────────────────────────────────────────

"ERROR: could not decrypt any key file (wrong password?)"
   -> The password is wrong, OR the password contains characters
      CMD mangled (rare; non-ASCII or quote characters).  Try the
      PowerShell entry point above or the LCSAS_PWFILE workflow.

"tree blob not found: <hex>"
   -> A pack file containing the blob is on a different disc.
      The next version of restore.bat will prompt for disc swaps;
      until then, mount all discs into one combined folder
      (xcopy /e D:\repo\data C:\merged\repo\data; xcopy /e
      E:\repo\data C:\merged\repo\data; ...) and point restore.bat
      at that combined folder.

"pack not found: <hex>"
   -> Same as above -- need the disc with that pack.

"ERROR: bad zstd blob size"
   -> Data corruption on the disc.  Try a different copy of the
      disc.  LCSAS archives are normally burned with redundant
      copies; one should be readable.

────────────────────────────────────────────────────────────────────
COMMAND-LINE REFERENCE
────────────────────────────────────────────────────────────────────

For users comfortable with CMD/PowerShell, restore.bat is just a
wrapper around lcsas-restore.exe.  The .exe accepts:

    --repo DIR             path to the restic repository
    --password-file FILE   path to a file containing the password
    --target DIR           where to write restored files
    --snapshot ID|latest   which snapshot (default: latest)
    --list-snapshots       print all snapshots; do not restore
    --verbose              print progress messages
    --catalog FILE         optional path to LCSAS catalog.db

Example:

    cd D:
    bin\x86_64-windows\lcsas-restore.exe ^
        --repo D:\repo ^
        --password-file C:\path\pw.txt ^
        --target C:\Users\me\restored ^
        --verbose

────────────────────────────────────────────────────────────────────
ON OLDER WINDOWS (XP / Vista / unpatched 7-8)
────────────────────────────────────────────────────────────────────

If lcsas-restore.exe fails to launch with an error mentioning a
missing api-ms-win-crt-*.dll, your Windows lacks UCRT.  Options
in order of effort:

  A. Install Windows Update KB2999226 (Sep 2015) on Windows 7/8/8.1.
     This adds UCRT.  Most patched systems already have it; if
     yours doesn't, run Windows Update and let it apply pending
     updates.

  B. Move the discs to a Windows 10 or 11 machine.  No installation
     of LCSAS is needed; just plug in the BD reader and double-click
     restore.bat on the newer machine.

  C. Use a Linux/macOS machine instead.  Boot a Linux live USB
     (Ubuntu, Mint, Debian -- any modern distro will do), insert
     the disc, and run:
        sh /media/<user>/<DISC>/restore.sh ~/restored
     The Linux binary has NO UCRT dependency; it only needs glibc
     or musl, both universal on Linux.

  D. Use the Python fallback.  Install Python 3 from python.org
     (versions 3.6 through 3.12 all work), then:
        py D:\standalone_restorer.py D:\repo C:\Users\me\restored
     This is tier 3 of the recovery cascade and works on any
     Windows that still runs Python 3.

  E. Future: an msvcrt-linked .exe.  See recovery/docs/
     WINDOWS_RECOVERY_PLAN.txt section "Phase W5 -- legacy msvcrt
     build" for the planned long-term solution that would also run
     on Windows XP / Vista.  Not yet available.

────────────────────────────────────────────────────────────────────
PHYSICAL DISC PROBLEMS
────────────────────────────────────────────────────────────────────

If the disc itself is unreadable (cracks, scratches, drive errors):

  1. Try a different optical drive (drives differ in how aggressively
     they retry weak sectors).
  2. The discs include DVDisaster RS03 error-correction data.  On
     Windows, install dvdisaster from https://dvdisaster.jcea.es
     (or your distro's equivalent), then:
        dvdisaster --read --image=disc.iso
        dvdisaster --fix --image=disc.iso
     Then mount the recovered .iso via Windows' built-in mount
     (right-click -> Mount) and re-run restore.bat against the
     mounted drive.
  3. If you have multiple copies of the same volume, any single
     readable copy is sufficient.  The on-disc binaries and packs
     are bit-identical across copies.
