LCSAS RECOVERY -- TIER STRUCTURE
=================================

WHY TIERS?

The recovery cascade is layered so that if any single component fails
(binary won't run, language interpreter is missing, etc.), the next
tier still gets you to the same outcome: plaintext files on disk.

Tiers 1-2 are the BARE MINIMUM PATH.  They share zero dependencies
beyond a kernel and a static libc baked into the binary.  If your
future system has those two things and one working CPU, the data is
recoverable.

Tier 3 is the OPTIONAL OVERFLOW.  It only runs if every C-based
option failed AND the user has not opted out via
LCSAS_ALLOW_PYTHON_TIER=0.

THE TIERS

  ┌────┬─────────────────────────────┬────────────────────────────┐
  │ #  │ Implementation               │ Runtime dependencies        │
  ├────┼─────────────────────────────┼────────────────────────────┤
  │ 1  │ Prebuilt lcsas-restore       │ kernel + libc (static)      │
  │    │ bin/<arch>/lcsas-restore     │ (musl baked in)             │
  ├────┼─────────────────────────────┼────────────────────────────┤
  │ 2  │ Vendored rustic-static       │ kernel + libc (static)      │
  │    │ bin/<arch>/rustic-static     │                             │
  ╞════╪═════════════════════════════╪════════════════════════════╡
  │ B  │   BARE MINIMUM PATH ENDS HERE                             │
  ╞════╪═════════════════════════════╪════════════════════════════╡
  │ 3  │ Pure-Python standalone_      │ python3 (>=3.7)             │
  │    │ restorer.py                  │ stdlib only (no pip needed) │
  └────┴─────────────────────────────┴────────────────────────────┘

DESIGN INVARIANT

The bare-minimum path (tiers 1-2) is verified Python-free by
``tests/test_bare_path.sh``:

  1. Statically inspect ``scripts/restore.sh`` -- the slice from
     "Tier 1:" to "BARE-MINIMUM PATH ENDS HERE" must contain no
     ``python`` or ``.py`` reference (in executable code; comments
     are skipped).

  2. Build a synthetic restic repo.  Strip every python interpreter
     from PATH by prepending a shim directory that exits 127 for
     anything named ``python*``.  Set ``LCSAS_ALLOW_PYTHON_TIER=0``.
     Invoke ``scripts/restore.sh``.  Verify recovery succeeds.

  Run as: ``make test-bare-path``

OPTING OUT OF TIER 3

If you want a hard guarantee that the recovery never touches Python:

  LCSAS_ALLOW_PYTHON_TIER=0 sh /restore.sh /mnt/recovery /tmp/restored

restore.sh will refuse to try tier 3 even if python3 and
standalone_restorer.py are both present.

TIER-2 ON MULTI-DISC ARCHIVES

Tier 2 (rustic-static) skips itself entirely on multi-disc archives
because rustic cannot drive the LCSAS disc-swap protocol -- it
expects every pack to live under a single ``$REPO/data/`` tree.
When restore.sh detects the multi-disc layout (no ``data/`` subdir
on the meta-disc-resident repo), it prints a one-line skip
diagnostic and falls through directly to tier 3.

On a multi-disc archive the effective cascade is therefore
TIER 1 -> TIER 3; tier 2 is a single-disc-only fallback.
Operators with a multi-disc archive should treat tier 1 as the
primary AND the only practical pre-Python fallback.

TIER BINARY PRE-FLIGHT

Before dispatching, restore.sh pre-flights each tier binary with
a size check plus a ``--help`` dry-run (see ``bin_preflight_ok`` in
``scripts/restore.sh``).  A zero-byte file or a wrong-arch binary
fails pre-flight and is skipped with the diagnostic:
    [tier N] ... is present but failed pre-flight ... skipping
The cascade then advances to the next available tier.  This guard
exists because POSIX ``sh`` execs an empty file as a no-op script
that exits 0 -- without the pre-flight, a bit-rotted tier-1 binary
would silently claim success and the operator would get nothing.

See ``RECOVER.txt`` (CASCADE DETAILS, pre-flight paragraph) for the
operator-facing version of this note.

TIER FALLBACK ON RUNTIME FAILURES

The default cascade ``exec``s the first available tier, so a tier-1
crash mid-restore aborts the run with no automatic fall-through.
This matches the bare-minimum recovery story: tier 1 IS the
recovery, and it has to work.

To make restore.sh walk the cascade on a tier-1 runtime failure
(non-zero exit), set ``LCSAS_TIER_FALLBACK=1``.  Tiers 1 and 2 then
run as subprocesses and fall through to the next tier on a non-zero
exit code.  Tier 3 is always ``exec``'d (last resort, nothing to
fall back to).  See ``RECOVER.txt`` ENVIRONMENT VARIABLES.

WHY PYTHON IS NOT ON THE BARE PATH

Python is a healthy ecosystem today, but at a 50-year horizon:

  * The CPython interpreter has fully replaced its parser, garbage
    collector, ABI, and standard library multiple times in 30 years.
    Source written today may not parse under a hypothetical Python
    of 2076.
  * Python's stdlib provides scrypt and HMAC -- but the
    standalone_restorer also requires dataclasses, ``from __future__
    import annotations`` semantics, and pathlib edge cases that
    pin the minimum at Python 3.7.  That single ecosystem version
    pin is itself a risk.
  * The interpreter binary on the recovery medium depends on
    glibc / musl + dozens of optional shared libraries (ssl, expat,
    libffi, etc.).  Static-link is possible but uncommon.

Statically-linked C binaries have neither of those properties.  The
prebuilt artefacts in tiers 1-2 link against musl baked into the
binary; the crypto primitives are spec-compliant per FIPS and IETF
documents that have not changed since they were published.

So: Python stays bundled as a redundant overflow path, BUT no part
of the workflow that we expect to actually run in 2076 requires it.

WHY TIER 1 IS THE PRIMARY (AND HOW WE COUNT DEPENDENCIES)

The cascade's design intent is that tier 1 -- our own lcsas-restore
binary built from C89 source against vendored sqlite + zstd -- is
the PRIMARY recovery tool.  Tier 2 (upstream rustic-static) exists
as a hedge in case our binary won't run for some reason on a
particular host; tier 3 (Python) is an emergency overflow.

"Vendored" matters here.  We distinguish two senses of "third
party":

  * VENDORED SOURCE -- we ship the C source for sqlite + zstd inside
    recovery/vendored/, compile it ourselves alongside our own code,
    and pin it by SHA-256 in MANIFEST.sha256 (the files-we-author
    manifest).  The build is reproducible from those sources.  An
    auditor can read every line that ends up in the lcsas-restore
    binary.  Vendored source is acceptable on the bare path.

  * RUNTIME DEPENDENCY -- a binary or interpreter we did not build,
    supplied by a third party (rustic; CPython) and distributed as
    an opaque prebuilt artefact.  Pinned upstream by SHA-256 in
    UPSTREAM.sha256, but the source it was compiled from is whatever
    upstream's CI did.  Runtime dependencies are explicit fallbacks
    (tiers 2 and 3); the bare path stays clean of them.

The intent: a recipient with nothing but a working CPU, a kernel,
a libc, and the lcsas-restore binary off the meta-volume should be
able to recover.  No `pip install`, no package manager, no upstream
release matrix that still needs to exist in 2076.  That's why
tier 1 deserves the cross-compile work even though Phase 21 shipped
the tier-2 cross-bundle as a shortcut -- see
docs/CROSS_PLATFORM_META_RFC.md §6 Q6.

WHY THE REBUILD-FROM-SOURCE TIERS WERE REMOVED

Earlier revisions of this cascade included two extra tiers between
the prebuilt binaries and the Python fallback:

  * Rebuild lcsas-restore from C source with whatever compiler is on
    PATH (cc/gcc/clang/tcc/pcc) plus POSIX make.
  * Rebuild rustic from vendored Rust source via cargo.

Both required a recipient who has a working toolchain (C or Rust)
but does NOT have the prebuilt static binaries shipped on the same
disc AND refuses to fall back to Python.  That intersection is
effectively empty: anyone with a C compiler can run a static
binary, and anyone with Python can run the pure-Python restorer.
Removing the rebuild tiers shrinks the cascade from 5 layers to 3
without sacrificing any reachable recovery scenario.

DEPENDENCY CHAIN AT A GLANCE

  Tier 1: musl libc -> CPU
  Tier 2: musl libc -> CPU
  Tier 3: python3 -> libc -> CPU

  The deepest bare-minimum chain is tier 1 or tier 2: 2 dependencies,
  each individually replaceable.

INVOCATION

  # Auto-discover recovery root from script location:
  sh /restore.sh /tmp/restored

  # Explicit:
  sh /restore.sh /mnt/recovery /tmp/restored latest

  # Skip Python tier:
  LCSAS_ALLOW_PYTHON_TIER=0 sh /restore.sh /mnt/recovery /tmp/restored

  # Non-interactive (password from env or file):
  LCSAS_PASSWORD='secret' sh /restore.sh /mnt/recovery /tmp/restored
  LCSAS_PWFILE=/path/pw    sh /restore.sh /mnt/recovery /tmp/restored
