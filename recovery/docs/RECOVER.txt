LCSAS RECOVERY TOOLCHAIN -- MANUAL RECOVERY PROCEDURE
=======================================================

PURPOSE

This document describes how to restore files from an LCSAS optical
archive using only the tools bundled on the recovery medium.  It
assumes nothing about the recovery environment beyond a working
POSIX-like OS with shell access.

PREREQUISITES

You will need:
  - The LCSAS recovery medium (BD-R / DVD).
  - The repository password (set when the archive was created).
  - A target directory with enough free space for the restored files.

DECISION FLOW

  Q: Do you have a working OS?
     NO  -> Boot the recovery medium directly.  See BOOT.txt.
     YES -> Continue.

  Q: Can you mount the recovery medium as an ISO9660 filesystem?
     NO  -> See "PHYSICAL RECOVERY" below.
     YES -> mount it read-only:

         mount -o loop,ro /dev/sr0 /mnt/recovery       (block device)
       OR
         mount -o loop,ro path-to-iso.iso /mnt/recovery (ISO file)

  Q: Does /mnt/recovery contain bin/<arch>/lcsas-restore for your arch?
     YES -> primary path:

         sh /mnt/recovery/scripts/restore.sh /mnt/recovery /tmp/restored

       This prompts for the password and writes restored files under
       /tmp/restored.  The script auto-detects architecture and falls
       back through the cascade if the primary binary is missing.

     NO  -> fall back to the pure-Python restorer (Tier 3):

         python3 /mnt/recovery/../standalone_restorer.py \
             /mnt/recovery/repo /tmp/restored \
             --password-file /tmp/pwfile

CASCADE DETAILS

The restore.sh script tries these in order.  Tiers 1-2 share NO
Python dependency; the BARE MINIMUM recovery path ends after tier 2.

  Tier 1.  bin/<arch>/lcsas-restore    prebuilt static C89 binary
                                       (fastest; depends only on
                                       musl libc + kernel)

  Tier 2.  bin/<arch>/rustic-static    vendored Rust binary
                                       (cross-check against tier 1)
  ─────────────────────────────────────────────────────────────────
  ^^ BARE MINIMUM PATH (zero Python).  If your future system has a
     working CPU and a kernel that can load a static binary, the
     data is recoverable.
  ─────────────────────────────────────────────────────────────────

  Tier 3.  standalone_restorer.py      pure-Python LAST RESORT
                                       (only invoked if every C/Rust
                                       option failed; skip entirely
                                       with LCSAS_ALLOW_PYTHON_TIER=0)

If all three fail, you have a hardware/environment problem.  See
"PHYSICAL RECOVERY" below.  See docs/TIERS.txt for the full design
rationale.

LIST AVAILABLE SNAPSHOTS

  /mnt/recovery/bin/<arch>/lcsas-restore \
      --repo /mnt/recovery/repo \
      --password-file /tmp/pw \
      --list-snapshots

Output:

  ID        TIME                          PATH
  a1b2c3d4  2026-01-15T10:30:00.000Z      /home/alice
  e5f6g7h8  2026-02-12T08:15:00.000Z      /home/alice

RESTORE A SPECIFIC SNAPSHOT

  /mnt/recovery/bin/<arch>/lcsas-restore \
      --repo /mnt/recovery/repo \
      --password-file /tmp/pw \
      --target /tmp/restored \
      --snapshot a1b2c3d4

The argument may be an exact ID or a unique prefix.

VERIFY YOUR RESTORE

Every blob is verified against its SHA-256 hash during restore.  If
any blob fails verification, lcsas-restore exits non-zero with a
message identifying the bad blob.  Re-run the restore against a
different disc copy (LCSAS maintains multiple copies of each volume).

PHYSICAL RECOVERY (CORRUPT MEDIUM)

If the disc returns I/O errors:

  1. Use ddrescue to make a best-effort copy:
         ddrescue /dev/sr0 /tmp/disc.iso /tmp/disc.log
     Repeat passes if /tmp/disc.log shows recovered sectors.

  2. The disc is augmented with DVDisaster RS03 ECC.  If you have
     dvdisaster installed:
         dvdisaster -i /tmp/disc.iso -f
     This restores up to ~30% of unreadable sectors.

  3. After ECC repair, mount the recovered ISO and continue with the
     normal cascade above.

PASSWORD RECOVERY

There is no password recovery path.  The password directly seeds the
scrypt KDF that derives the master key encryption key.  If you have
lost the password, the data is cryptographically unrecoverable.

If multiple key files exist under repo/keys/, lcsas-restore tries each
in turn; you only need ONE valid password.

DISC MIRRORING

If you have multiple copies of an LCSAS volume (as is the default),
they are bit-identical.  Any single working copy is sufficient.  To
verify which disc you are using:

  cat /mnt/recovery/volume_label.txt    # human-readable label
  sha256sum /mnt/recovery/MANIFEST.sha256

SUPPORT

This is offline, after-the-fact recovery.  There is no support phone
number, no vendor.  The instructions in this file, the C source under
src/, and the documents under docs/ are everything you need.

EXIT CODES

  lcsas-restore exits:
    0 -- success
    1 -- runtime error (bad password, missing file, integrity failure)
    2 -- usage error

MULTI-DISC RESTORE

An LCSAS archive typically spans multiple optical discs: one META disc
that carries the recovery toolchain and holographic catalog, plus
several DATA discs that hold the encrypted pack files.  The recovery
cascade handles swaps interactively — but the operator must know what
to expect and what to do.

WHICH DISC DO I START WITH?

Always start with the META disc — the one labelled LCSAS_META.  It is
the only disc that contains the recovery binaries (bin/<arch>/), the
catalog, and restore.sh.

  1. Insert the META disc and mount it:
         sudo mount /dev/sr0 /mnt
  2. Run:
         sh /mnt/scripts/restore.sh ~/restored/
  3. restore.sh copies itself and the binary into RAM (/tmp) so the
     META disc can later be ejected.  You will see:
         [lcsas-restore] copied recovery binaries to /tmp/lcsas-restore.Xxxx
         [lcsas-restore] you may eject the recovery disc when the
                         binary prompts for a data disc.
  4. When the binary has loaded the index and needs a pack that lives
     on a DATA disc it pauses and prints a prompt (see below).

WHAT THE SWAP PROMPT LOOKS LIKE

When lcsas-restore cannot find the next required pack on the currently
mounted disc it prints (to stderr):

  +----------------------------------------------------------+
  | Pack ... is required for the next file.                  |
  | ...                                                      |
  | Single-drive recovery: if your machine has only ONE      |
  | optical drive, eject the RECOVERY disc first, then       |
  | insert the disc named above into the SAME drive.         |
  | Insert the right disc and press ENTER to retry.          |
  +----------------------------------------------------------+

The catalog hint lines will name the specific DATA disc (e.g. "vol-042
Family Photos 2027 vol 3/8") when a catalog.db is available on a
mounted disc.  If no catalog is present the prompt shows only the pack
hash and asks you to try each DATA disc in turn.

PHYSICAL SWAP STEPS (single optical drive)

  1. Note the disc label named in the prompt.
  2. Eject the current disc:
         eject /dev/sr0
     (On macOS: drutil eject)
  3. Insert the DATA disc named in the prompt.
  4. Wait for the OS to auto-mount it (typically a few seconds).
     If auto-mount is disabled, mount it manually:
         sudo mount /dev/sr0 /mnt
  5. Press ENTER at the prompt.
  6. lcsas-restore rescans the mount directories and continues.
     If the wrong disc was inserted it will re-prompt; eject and
     retry with the correct disc.
  7. Repeat for each additional DATA disc as prompted.

NOTE ON THE TEST RIG vs. REAL OPERATORS

Automated tests use a setuid C wrapper called disc-loader to simulate
disc swaps without human interaction.  Real operators must perform the
eject/insert step manually.  A passing test therefore does NOT guarantee
that the UX is smooth for a live operator — this section exists because
the test rig hid the gap for some time (see UX_CONCERNS.txt ID 011).

EXPECTED SWAP COUNT (FIRST-CONTACT COST)

Even with the pack cache enabled, the first contact with each DATA disc
always requires a physical swap — one swap per disc, minimum.  For an
archive spread across N data discs, expect at least N swap prompts.
With the cache active each disc is needed only once (O(N) total swaps);
subsequent sessions find all packs in cache and need no disc at all.
Without the cache, the same disc may be requested multiple times within
a single restore session.

REDUCING SWAP FREQUENCY WITH THE PACK CACHE

By default restore.sh allocates an opportunistic pack cache under /tmp
(LCSAS_PACK_CACHE_DIR=auto).  When lcsas-restore first reads a pack
from a DATA disc it copies the entire data/ subtree of that disc into
the cache.  Subsequent packs from the same disc are served from the
local cache, not the physical medium, so each DATA disc typically needs
to be inserted only ONCE regardless of how many packs it contributes
to the restore.

  * To use a specific directory:   export LCSAS_PACK_CACHE_DIR=/path
  * To disable (disk-constrained):  export LCSAS_PACK_CACHE_DIR=
  * Disk cost: roughly equal to the total size of all DATA discs that
    are touched during the restore (worst case: the full archive).

IF YOU HAVE ONLY ONE OPTICAL DRIVE

You do NOT need two Blu-ray drives.  restore.sh detects that it is
running from a read-only mount (the meta-disc), copies itself plus
the binary tree to a writable directory (/tmp by default), and
re-executes from there.  After this relocation, no file handles
remain on the meta-disc, so when the binary prompts for a data
disc you can eject the meta-disc and insert the data disc into
the same drive.

The prompt explicitly tells you when to swap:

  +----------------------------------------------------------+
  | Pack ... is required for the next file.                  |
  | ...                                                      |
  | Single-drive recovery: if your machine has only ONE      |
  | optical drive, eject the RECOVERY disc first, then       |
  | insert the disc named above into the SAME drive.         |
  | Insert the right disc and press ENTER to retry.          |
  +----------------------------------------------------------+

Set LCSAS_NO_RELOCATE=1 to suppress the relocation step (only
useful when running the script from a writable checkout for
development).
