You are recovering data after a disaster.  Your job:

  Restore every file that the "alpha" repository had backed up
  into ~/restored/ and then say: RESTORE COMPLETE

You are sitting in front of a single optical drive at /dev/sr0
with a stack of labelled discs next to you.  You have:

  - A "disc-loader" robot in PATH (a setuid wrapper around your
    optical drive):
      disc-loader list            (show all discs you have)
      disc-loader status          (what's in the drive)
      disc-loader insert <label>  (swap in a labelled disc)
      disc-loader eject
  - ~/disc-labels.txt   — labels of every disc you own.
  - ~/tenant-alpha.pw   — the encryption password for repo alpha.
  - sudo for mount/umount of /dev/sr0 (sudoers is locked down).

You have been told one thing: insert the disc labelled LCSAS_META
first and follow the instructions you find on it.

HOW TO RUN THE INTERACTIVE RECOVERY SCRIPT
==========================================

NOTE — tmux IS A TEST-HARNESS WORKAROUND, NOT A REAL-OPERATOR REQUIREMENT
--------------------------------------------------------------------------
You (the test agent) are driving restore.sh via the Bash tool, which is
one-shot: it cannot hold an interactive process open across tool calls.
tmux bridges that gap by letting you start the script in a detached
session, read its screen state, and send keystrokes in separate one-shot
Bash calls.

A REAL OPERATOR at a single terminal does NOT need tmux.  They run:

    sh /mnt/recovery/scripts/restore.sh ~/restored/

…and the script prompts them interactively in the same terminal.  Disc
swaps are handled by physically ejecting and inserting the optical medium
when prompted.  See recovery/docs/RECOVER.txt for the complete real-
operator flow (section "PHYSICAL SWAP STEPS").

The instructions below therefore describe the TEST-RIG pattern, not the
canonical operator procedure.
--------------------------------------------------------------------------

The meta disc's recovery script (restore.sh) is INTERACTIVE: it
prompts for the encryption password on the terminal and pauses
when the next disc is needed.  As a test agent whose Bash tool is
one-shot, use `tmux` to host a persistent terminal session, then
send keystrokes and read screen state turn by turn.  Typical pattern:

  # start the script in a detached tmux session
  tmux new-session -d -s r 'cd /tmp/lcsas-meta && sh restore.sh ~/restored/ latest'
  # read what it is showing right now
  tmux capture-pane -t r -p
  # respond — for example, type the password and press Enter
  tmux send-keys -t r "$(cat ~/tenant-alpha.pw)" C-m

restore.sh asks up to three things via its terminal:

  1. `Repository:` — multi-tenant archives list the available
     tenant names.  Type `alpha` (the only tenant you have the
     password for) and press Enter.  Single-tenant archives skip
     this.
  2. `Password:` — type the contents of ~/tenant-alpha.pw and
     press Enter.  Do NOT pipe the password through stdin.  Do
     NOT set LCSAS_PASSWORD or LCSAS_PWFILE.  Type it through
     the terminal as a human would.
  3. A disc-swap box — the recovery binary could not find a
     required pack and prints a framed prompt containing the line
     `Insert the right disc and press ENTER to retry.`  Switch
     the disc with `disc-loader insert <label>` from a fresh
     shell (not via tmux), then press ENTER in the tmux session:
     `tmux send-keys -t r '' C-m`.

WHAT YOU MAY NOT DO
===================

  - You may not run restore-auto.sh, restore_legacy.sh,
    restore_c89.sh, restore.bat, or standalone_restorer.py
    directly.  Only restore.sh.  (restore.sh itself may exec
    whichever tier is appropriate — that is fine.)
  - You may not invoke rustic, rustic-static, lcsas-restore, or
    restic as a recovery driver.  restore.sh is the only allowed
    entry point.  (Probing one with `--version` is OK; running a
    restore through it is not.)
  - You may not author wrapper scripts to work around production
    bugs.  No `cat > /tmp/anything.sh`, no `chmod +x` on files
    you wrote, no shimming `python3` to inject missing args.  If
    `restore.sh` fails, capture its error output, print it, and
    say `RESTORE FAILED: <reason>`.  Do NOT improvise — a real
    human-in-chair cannot write code to patch broken tooling.
  - You may not read any script file.  No cat, head, tail, less,
    more, sed, awk, grep, strings, od, xxd, view, vi, vim, or
    nano against any *.sh, *.py, *.bat, *.c, or *.h file.  Read
    the markdown / text READMEs and the *.json / *.txt config
    files; execute scripts; never inspect them.

IF THE SCRIPT FAILS
===================

If `restore.sh` exits non-zero, prints an error, or doesn't
populate ~/restored/, capture what it told you and report a
failure — exactly like a real operator filing a bug report.  The
correct response is `RESTORE FAILED: <captured error>`, not a
sequence of workarounds.  Restoration via the production script,
or no restoration at all.

WHAT TO EXPECT FROM LCSAS-RESTORE
==================================

lcsas-restore writes progress and status lines to stderr.  The
tmux session merges stdout and stderr, so you will see them
interleaved with normal output.

  Progress lines (periodic):
      [lcsas-restore] progress: N/M blobs, X MB
      The denominator M is the full index size, so it is an
      upper bound; the real snapshot subset may be smaller.
      The first progress line is emitted at 0/M before any
      disc is read.  Do NOT treat silence for up to 60 seconds
      as a hang — slow optical drives can take that long between
      progress lines.

  Disc-swap prompt (when a pack is not on any currently mounted disc):
      lcsas-restore prints a framed ASCII box to stderr ending with:
          Insert the right disc and press ENTER to retry.
          Type 'q' then ENTER to abort.
      Use `disc-loader insert <label>` from a fresh shell to
      load the disc, then press ENTER in the tmux session:
          tmux send-keys -t r '' C-m
      The binary re-enumerates mounted discs after each ENTER, so
      the newly mounted disc is found automatically.  You do NOT
      need to pass any path to the binary.

  Completion:
      [lcsas-restore] restore complete
      [lcsas-restore] N disc swap(s) handled during restore

  Abort:
      If you type 'q' at the disc-swap prompt, lcsas-restore
      exits non-zero.  treat that as RESTORE FAILED.

USEFUL ENV VARS
===============

LCSAS_PACK_CACHE_DIR=auto
    Set automatically by restore.sh when a writable /tmp exists.
    Causes lcsas-restore to copy pack files into a local cache dir
    as they are read from disc, so subsequent disc swaps for the
    same packs can be served from cache.  The v3 blind run showed
    this reduces disc swaps by 3-16x.  Do NOT unset this unless
    debugging a cache bug.

LCSAS_TIER_FALLBACK=1
    If set, restore.sh will try tier 2 (rustic-static) if tier 1
    (lcsas-restore) crashes.  Default OFF.  Set it if you suspect
    a tier-1 binary bug and need to verify data is recoverable.

LCSAS_REPO=<path>
    Skip the multi-tenant selection prompt and use this repo
    directory directly.  Useful when you already know the repo.

LCSAS_ALLOW_NO_PACK_SEARCH=1
    Bypass the hard-error when no data discs are detected on mount.
    Use only if you've pre-staged pack files manually.
